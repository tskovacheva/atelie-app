# Глина

**Малко приложение за керамично ателие.** По-близък дневник, отколкото инструмент — за рецепти на глазури, материали, експериментални тестове и животa на изделията от мокра глина до финал.

> Направено за собствена употреба в ателие, не за общ продукт.

[**→ Отвори приложението**](https://tskovacheva.github.io/atelie-app/)

## Какво прави

Шест раздела, които покриват цикъла на ръчната керамика:

- **Изделия** — каталог на парчетата с етапи (greenware → leather-hard → bone-dry → bisque → glazed → glaze-fired → finished), снимки, свързани глини и глазури
- **Рецепти** — глазурни рецепти със съставки и проценти, температура/конус, тип, препоръчан профил на изпичане, калкулатор за грамове при произволен батч
- **Материали** — склад на суровини и готови глазури с известия за ниски запаси, химичен анализ, wishlist за бъдещи покупки
- **Тестове** — дневник на глазурни опити с оценка, дефекти, резултатни полета, снимки
- **Изпичания** — profile-и с ramp rates, hold time, задръжка; свързани с тестове и рецепти
- **Библиотека** — лична база знания с предзаредени справочни материали

Всеки раздел поддържа търсене, филтри, сортиране и дублиране на записи. Между тях има **свързани записи** (derived relations) — от рецепта виждаш свързаните изделия и тестове, от профил виждаш кои тестове и рецепти го ползват, и т.н.

## За кого е

Ателие-базирана керамична практика, не търговски поточен процес. Ако правиш серийно производство, продаваш партиди или водиш клиенти — това не е продуктът за теб.

## Как работи

- **Single-file PWA** — един HTML файл, `sw.js`, manifest, икони. Без build step.
- **Без backend, без акаунти.** Всички данни се пазят локално в браузъра (localStorage).
- **Работи офлайн** благодарение на service worker.
- **Инсталира се като app** на телефон или десктоп (Add to Home Screen / Install App).
- **Backup/restore през JSON** — в раздел Библиотека, долу. Препоръчителен workflow: експорт след всяка сесия, с дата и устройство в името.

## Технически детайли

- Vanilla JavaScript (ES5+ syntax), inline CSS, inline HTML. Без framework-и, без dependencies.
- Около 2500 реда общо. Оптимизирано за четливост, не за производителност.
- Data structure: `{pieces, recipes, materials, tests, firingProfiles}` в localStorage под ключ `atelie_v6`; библиотеката отделно под `atelie_userlib`.
- Derived relations — никакви reverse foreign keys не се пазят; всички „свързани" списъци се изчисляват при отваряне на detail екран.
- Service worker cache strategy: navigation → network-first, assets → cache-first. `CACHE_NAME` се bump-ва при всеки deploy.
- Design system от v1.2.0: sage + terra cotta палитра, Tenor Sans + Jost шрифтове. Стари CSS variables запазени като aliases за backward compatibility.

## Запазване на данни — важно

Понеже всичко е в localStorage:

- **Данните са само на това устройство.** Ако работиш на телефон и лаптоп — правиш експорт и импорт ръчно.
- **При деинсталиране на PWA** може да се изтрият данните (зависи от OS). Винаги експортвай преди reinstall.
- **Между устройства** няма автоматична синхронизация. Workflow: експорт от A → качване в облак → сваляне на B → импорт. При импорт на по-стар файл, приложението предупреждава.
- **Backup файлове** съдържат timestamp + device в името (напр. `atelie_2026-04-20_1435_mobile.json`), за да не се губи ред при няколко устройства.

## Deploy

Върху GitHub Pages. Пет файла за production:

```
atelie_v6.html
sw.js
app.webmanifest
icons/icon-192.png
icons/icon-512.png
```

`ROADMAP.md` и `README.md` живеят в repo-то, но не се сервират директно на потребители.

При всеки production update:
1. Bump `APP_VERSION` в `atelie_v6.html`
2. Bump `CACHE_NAME` в `sw.js`
3. Commit + push → GitHub Pages deploys автоматично

## Принципи

- **Малки patch-ове.** Инкрементални промени без refactor.
- **Backward compatibility.** Всеки patch трябва да работи със стари localStorage данни.
- **Safe guards навсякъде.** Null checks, `|| ''` fallbacks, optional chaining за липсващи полета в стари records.
- **Derived, не дублирано.** Свързаните записи се изчисляват, не се пазят.
- **Без външни dependencies.** Един file, без build tooling.
- **UX над features.** Ако feature не решава реална ежедневна болка, не се добавя.
